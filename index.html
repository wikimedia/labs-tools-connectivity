motivation

MediaWiki collects two lists: orphaned and deadend pages. It is important to understand what the difference between page and article is. Page is everything in main namespace, which is not a redirect, including disambiguation pages. The page is linked when any link from other page points to the page we are considering. Even when this link is from other namespace (page has been considered to be deleted e.g. or just a discussion about this page taken place somewhere) or from disambiguation page (disambiguations are to be unlinked, so links from them do not help connecting articles one to another).

MediaWiki also does not recognise links from lists or chronological articles. Such links are rarely relevant and nobody usually came from them. Other rules can also be introduced when we need to understand how well connected our articles are.

Thinking about connectivity we may need to consider more than just orphanes. There may be two, three or more pages connecting to each other but without any links from outside of a group, i.e. isolated clusters (or strongly connected components). One cluster may link other and there may be a chain of isolated clusters, which is not linked from the main connected component containing most part of good articles. MediaWiki does not recognize isolated articles at all.

When MediaWiki locates dead-end pages it looks if the page contain a link. It doesn't matter for the engine if this link is set from a template and just talks about some problems in a page or the link is to an article. Links to disambiguation pages are also threated as valid by the engine.

Connectivity analysis for articles allows to authors make their articles better adding them an attention of other people, who are authors of other articles or just readers following links.

One more point to begin with connectivity analysis is that this task is usually solved in with procedural languages analyzing offline links bases. In order to make the connectivity data more actual it is necessary to run the analyzer closer to severs to avoid large data transmittal. So, it is required to be written in SQL, and there were no SQL solutions for oriented graph connectivity analysis introduced. So, here we go.

scope

Let suppose we have an article defined as non-redirect and non-disambiguation page from the main namespace. Let also suppose article has a link from other article if this is a direct or redirected link from non-chronological article (can be easily disabled if necessary). Let suppose the article is not a dead-end page if it contains one or more links to other existent articles. The following tasks are now solved:
  • Dead-end articles list according to the above definition
  • Orphaned articles list according to the above definition
  • All types of isolated clusters and chains of isolated clusters, which are present and have size less than some threshold
Tasks solved by the way:
  • List of double and triple redirects, which for unknown reason is more complete than the list MediaWiki produces
  • List of wrong redirects (having uncommented text with links after redirection magic.
  • Statistics for isolated clusters amount for different types of clusters
  • Data for querying isolated articles for each particular category (see the bottom of this page)

how does it work

There are two mains scrips. One is isolated.sh, it runs on hemlock to drive the process. The second script isolated.sql is called by isolated.sh, works yarrow (for ruwiki at the moment) and does all the connectivity analysis. When some sort of articles is found, the sql scripts puts some command to stdout, which is controlled by isolated.sh, and switches the output handler to put data to a file, upload it to Wikipedia or do something specific.
